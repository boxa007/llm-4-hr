# Онбординг: Контент-менеджер · Profigent

> Внутрішній документ для нових співробітників на позиції Контент-менеджер.
> Джерело правди. Якщо щось у процесі змінилось - оновіть цей файл, а не передавайте усно.
> Питання, не покриті тут → people@profigent.ai або у HR-бота.

---

## 1. Хто ти і за що відповідаєш

Контент-менеджер у Profigent веде LinkedIn-контент клієнтів-фаундерів. Profigent - це GTM-агентство: ми будуємо клієнтам вхідний потік через LinkedIn (контент + outreach). Ти - не копірайтер «напиши пост», ти контракт-контент-менеджер: відповідаєш за весь цикл клієнта від брифу до опублікованого поста.

**Твоя зона відповідальності:**
- Голос клієнта (voice-profile) - щоб пости звучали як він, а не як AI
- Місячний контент-план (12 постів) - бій «біль × етап воронки»
- Написання постів через систему (Claude Code + скіли), не вручну з нуля
- 3 гейти узгодження з клієнтом: бриф → план → пост-з-5
- Якість: кожен пост проходить fact-check і quality-check перед видачею

**За що ти НЕ відповідаєш:** дизайн банерів робиться через скіли (не вручну), outreach веде окрема роль, технічну підтримку системи - команда продукту.

## 2. Перший тиждень: що зробити по днях

**День 1.**
- Отримай доступи: Obsidian-vault (через people@profigent.ai), Claude Code, n8n (app.profigent.ai), Notion, Buffer
- Прочитай `CLAUDE.md` у корені vault - це інструкція всієї системи
- Прочитай `MANAGER-HANDBOOK.md` - твій головний walkthrough
- Подивись 3 навчальні лекції у `clients/profigent/lectures/`

**День 2-3.**
- Розберись зі структурою клієнта: `clients/[slug]/profile/`, `content-plan/`, `drafts/`
- Прочитай brief.md одного активного клієнта повністю
- Прочитай `_system/voice-system.md` - як працює клонування голосу (критично)
- Зроби тестовий пост через `/write-post` на демо-клієнті, прогони до quality-check

**День 4-5.**
- Тебе закріплюють за 1 реальним клієнтом (shadow-режим: робиш, ментор перевіряє)
- Збери для нього reference-mining перед постом (`/research ideas`)
- Напиши 1 пост повністю, проведи через гейт пост-з-5

**Кінець тижня 1.** Ти вмієш: прочитати бриф, зібрати ресёрч, написати пост у голос клієнта, провести через quality-check і fact-check.

## 3. Робочі інструменти (стек)

| Інструмент | Для чого | Де доступ |
|---|---|---|
| **Obsidian-vault** | вся база клієнтів, плани, чернетки | people@profigent.ai видає |
| **Claude Code** | генерація постів, ресёрч, скіли | особистий акаунт |
| **n8n** (app.profigent.ai) | автоматизації, генерація картинок | спитати у тімліда |
| **Notion** | плани для клієнтів, КП | командний воркспейс |
| **Buffer** | публікація і метрики постів | per-client токен у buffer.md |
| **Apify** | ресёрч (скрейп LinkedIn/Reddit/X) | через скрипти, токен авто |

**Правило стека:** ми не пишемо пости вручну в Google Docs. Усе через систему - так зберігається голос, fact-check і метрики. Ручний пост = пост без перевірок = ризик.

## 4. 3 гейти узгодження (святе правило)

Жоден пост не виходить клієнту без проходження гейтів:

1. **Гейт 1 - Бриф.** Поки `brief_approved=false` у `_status.md`, не можна робити план. Бриф узгоджує клієнт.
2. **Гейт 2 - План.** 12 постів на місяць → клієнт затверджує → тільки потім генерація.
3. **Гейт 3 - Пост-з-5.** На кожен слот система дає 5 кандидатів → клієнт (або ти за нього за домовленістю) обирає 1.

**Чому це важливо:** ми контракт-менеджери, а не «фабрика тексту». Клієнт має контроль на 3 точках. Пропустити гейт = підірвати довіру клієнта.

## 5. Стандарти якості (must-know)

- **Голос клієнта обов'язковий.** Кожна генерація читає `voice-profile.json`. Немає файлу → СТОП, запустити `extract-voice.sh`. Generic-AI пости заборонені.
- **Жодного м-дешу (- чи -).** Тільки звичайний дефіс з пробілами. Це жорстке правило власника. Перевір перед видачею.
- **Реальні цифри.** У пості мінімум 1 конкретне число (цифрою 0-9, не прописом). Вигадані кейси заборонені.
- **Fact-check перед «готово».** Кожен світовий факт (ринкова цифра, дата, чужа компанія) перевіряється живим пошуком в інтернеті. Без цього пост неповний.
- **Reference-mining перед КОЖНИМ постом.** Топ-3 реальні пости клієнта + buyer-corpus інжектяться у генерацію. Не пишемо «з голови».

## 6. Графік, комунікація, очікування

- **Робочий день:** гнучкий, але слоти постів за планом мають виходити вчасно. Орієнтир - результат, не години.
- **Стендап:** короткий синк з тімлідом (формат і час - уточни у тімліда в перший день).
- **Реакція на клієнта:** правки клієнта по чернетці - в той самий день фіксуємо у `corrections.md` цього клієнта (це memory-шар, що вчить систему).
- **Ескалація:** застряг технічно → тімлід. Питання по клієнту → акаунт-менеджер. HR-питання → people@profigent.ai.

## 7. Випробувальний термін: критерії

Перші 30 днів. Очікуємо, що до кінця ти:
- Самостійно ведеш 1-2 клієнтів повний цикл (бриф → план → пост)
- Тримаєш голос клієнта (пости проходять voice-audit без переробок)
- Не порушуєш гейти і стандарти якості
- Фіксуєш правки клієнта у corrections (не повторюєш ту саму помилку двічі)

Зворотний зв'язок - на 14-й і 30-й день з тімлідом.

## 8. Часті питання новачка

**«Скільки клієнтів я веду?»** Старт - 1-2 у shadow. Далі залежить від швидкості, зазвичай 3-5 активних.

**«Що, якщо я не знаю голос клієнта?»** Голос не «вгадується» - він у `voice-profile.json`, зібраний з реальних постів. Читаєш файл, не вигадуєш.

**«Можна писати пост вручну, якщо швидше?»** Ні. Втрачаються голос, fact-check і метрики. Завжди через систему.

**«Клієнт просить пост терміново, без ресёрчу - можна?»** Терміновість не скасовує reference-mining перед постом. Це стандарт. Швидко ≠ без перевірок.

**«Де брати ідеї для постів?»** З якорів: voice z-score (що залітало), interviews, brief, ресёрч. Не «з позиціонування під настрій».
